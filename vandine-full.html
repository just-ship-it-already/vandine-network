NASA JPL Power of 10 Rules

Developed by Gerard J. Holzmann at NASA's Jet Propulsion Laboratory, these rules were created for writing safety-critical code that powers Mars rovers, deep space probes, and other mission-critical systems. Here's how each principle is applied in the Vandine infrastructure:

1 Restrict to Simple Control Flow

Avoid complex control flow constructs like goto, setjmp/longjmp, and direct or indirect recursion. This makes the code more predictable and easier to analyze with static analysis tools.

// ‚ùå Bad: Recursive function
function factorial(n) {
  return n <= 1 ? 1 : n * factorial(n - 1);
}

// ‚úÖ Good: Iterative approach
function factorial(n) {
  let result = 1;
  for (let i = 2; i <= n && i <= MAX_FACTORIAL; i++) {
    result *= i;
  }
  return result;
}
Why This Matters:

In space missions, unpredictable code paths can cause catastrophic failures. Simple control flow ensures every execution path can be tested and verified. The Mars Climate Orbiter loss in 1999 highlighted the importance of predictable, verifiable code.

2 Give All Loops a Fixed Upper Bound

It must be possible to prove statically that all loops will eventually terminate. This prevents infinite loops that could freeze critical systems.

// ‚ùå Bad: No upper bound
while (sensor.isActive()) {
  processData();
}

// ‚úÖ Good: Fixed upper bound
const MAX_ITERATIONS = 1000;
let iterations = 0;
while (sensor.isActive() && iterations < MAX_ITERATIONS) {
  processData();
  iterations++;
}
Why This Matters:

The Spirit rover on Mars once got stuck in an infinite loop, causing a critical delay. Fixed bounds ensure systems can recover and continue operation even if unexpected conditions arise.

3 Do Not Use Dynamic Memory Allocation After Initialization

Heap memory allocation (malloc, new) should only happen during initialization. This prevents memory fragmentation, leaks, and unpredictable allocation failures.

// ‚ùå Bad: Dynamic allocation during runtime
function processRequest(data) {
  const buffer = new Array(data.length);
  // Process...
}

// ‚úÖ Good: Pre-allocated pools
const BUFFER_POOL = new Array(MAX_BUFFERS);
function init() {
  for (let i = 0; i < MAX_BUFFERS; i++) {
    BUFFER_POOL[i] = new Array(BUFFER_SIZE);
  }
}
Why This Matters:

Memory allocation failures have caused numerous spacecraft anomalies. Pre-allocation ensures predictable memory usage and prevents fragmentation that could occur over months or years of continuous operation in space.

4 No Function Should Be Longer Than 60 Lines

What fits on a single printed page can be understood as a unit. Shorter functions are easier to test, debug, and formally verify.

// Functions broken down into manageable units
function validateTelemetry(data) { /* max 60 lines */ }
function processTelemetry(data) { /* max 60 lines */ }
function storeTelemetry(data) { /* max 60 lines */ }

// Each function has a single, clear responsibility
Why This Matters:

Complex functions hide bugs. NASA's code review process found that functions over 60 lines had 3x more defects. Smaller functions enable comprehensive testing and peer review.

5 Use a Minimum of Two Runtime Assertions Per Function

Assertions check assumptions and catch errors early. They serve as executable documentation of expected conditions.

function calculateTrajectory(velocity, angle) {
  assert(velocity > 0 && velocity <= MAX_VELOCITY);
  assert(angle >= 0 && angle <= 360);
  
  const result = /* calculation */;
  
  assert(isFinite(result));
  assert(result >= MIN_TRAJECTORY && result <= MAX_TRAJECTORY);
  return result;
}
Why This Matters:

Assertions saved the Mars Pathfinder mission when they detected an priority inversion issue. They provide runtime verification that catches issues traditional testing might miss.

6 Restrict Data Scope to the Smallest Possible

Variables should be declared in the smallest scope possible and as close to their first use as possible. This minimizes the state space and potential for errors.

// ‚ùå Bad: Wide scope
let temp, result, counter;
function process() {
  temp = getData();
  counter = 0;
  // ...
}

// ‚úÖ Good: Narrow scope
function process() {
  const temp = getData();
  let counter = 0;
  // Variables exist only where needed
}
7 Check Return Values of All Non-Void Functions

Every function that returns a value must have that value checked. Ignored return values are a common source of subtle bugs.

// ‚ùå Bad: Ignoring return value
file.write(data);
sensor.calibrate();

// ‚úÖ Good: Checking returns
if (!file.write(data)) {
  handleWriteError();
}
if (!sensor.calibrate()) {
  logCalibrationFailure();
}
8 Use the Preprocessor Sparingly

Limit preprocessor use to file inclusions and simple conditional compilation. Complex macros hide bugs and make debugging difficult.

// ‚ùå Bad: Complex macro
#define CALC(x,y) ((x)>0?(x)*(y):(x)+(y))

// ‚úÖ Good: Type-safe function
function calculate(x, y) {
  return x > 0 ? x * y : x + y;
}
9 Limit Pointer Use and Forbid Function Pointers

Pointers should be used sparingly. Function pointers are forbidden as they make control flow analysis nearly impossible.

// ‚ùå Bad: Function pointer
void (*operation)(int) = getUserFunction();
operation(data);

// ‚úÖ Good: Explicit control flow
switch (operationType) {
  case ADD: add(data); break;
  case MULTIPLY: multiply(data); break;
  default: handleError();
}
10 Compile with All Warnings Enabled; Treat Warnings as Errors

All code must compile cleanly at the highest warning level. Modern compilers can detect many potential issues - ignoring warnings is ignoring free bug detection.

// Compiler flags for zero-warning builds
// gcc: -Wall -Wextra -Werror
// TypeScript: "strict": true, "noImplicitAny": true
// Zero warnings = Zero compromises on code quality
Why This Matters:

A single ignored warning in the Ariane 5 software led to a $370 million failure. Treating warnings as errors ensures code meets the highest quality standards.
